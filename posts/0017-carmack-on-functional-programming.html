carmack on functional programming

Posted on September 30, 2014 by devty

I had always perceived functional programming as derivative of an academic exercise - immensely stimulating but practically of minute concern.

So it came to me as great surprise to discover John Carmack, you know THE John Carmack, making some comments on functional programming. This is a person who I had previously assumed to be coding in bits, doing things at the extremity of low-level programming.

In the end John Carmack is a software engineer, and he is plainly commenting on the benefits functional programming offers over the imperative style. He is not asking everyone to abandon c/c++ in favor of Lisp and Haskell but wants to utilize the strengths those non-mainstream languages exercises with its functional style.

A brief summary of his post:

  1. pure functions are nice And Have no side effects - it is a well-defined function (in mathematical sense) where a given input has one possible output regardless of the environmental condition. Composing a program with pure functions have nice benefits - thread safety, trivial transition to parallel implementation, reusability in new environment (as it is agnostic to the environment), testability since it is very simple to generate a test case to guarantee its behavior. Pure functions tends be stylish and rather easy to understand - it is usually a nicely defined unit with an acute intention. It also opens the door for formal verification. Pure functions are concrete enough to make general formal statement about it.

  2. real things can’t be pure And it doesn’t have to be. There is a continuum of value in purity of functions - moving towards purity and reaping the benefits is still better than completely entangled spaghetti code. Note that moving towards purity by disentangling functions from the environment may result in code verbosity - now you have to completely encapsulate the environmental variable and pass them as a parameter of your pure functions. Conciseness is often a characteristic of functional programming, and this verbosity may seem a bit curious. But the factors that renders functional programs succinct are often orthogonal to the use of pure functions - it’s the language’s built-in types, pattern matching, list comprehensions, function compositions, and syntactic sugars that are the greatest contributor to the compactness of code in those languages.

  3. OOP is inherently not functional Mutating object is the lesson 0 of in object oriented paradigm - and counter to the functional programming style. You can try, but it’s just weird.

  4. functional style may be slower but.. perhaps not. Yes directly mutating values in memory is faster, but consider two points. One we’ve always been willing to sacrifice few points on performance for productivity. Second, pure functions invites parallel programming - so not all is lost.

I’m pretty certain we’ll see more and more of functional paradigm entering the mainstream programming community.
