thinking about types

Posted on August 26, 2014 by devty

Types have been creeping up in my life from all kinds of direction.

It first stared with haskell when I saw that type annotation of a function, combined with a useful name, sometimes are enough to completely conceptually define a function (which sometimes eliminated a need for a documatation!). It made reasoning with programming an entertaining exercise of breaking the problem down to a collection of function signatures - not a novel concept but never more clear and easy as done in haskell.

Then came my reading of an open source program for Python - a dynamically typed language - which was hampered by the lack of type annotation I had come to appreciate in Haskell. Not knowing the type of input the function expects and what the intended output is to be made reading of the code a very different process - a lot of jumping around the source code, trying to see through the coupling of functions. First time I felt mistreated by Python’s dynamic type system. Perhaps this mistreat meant is not entirely types own fault - maybe it was purely functional aspect - the quality it of being side-effect free - of Haskell that I missed more dearly.

So came my interest in mypy, a statically typed variant of Python 3. It’s a really interesting idea - mypy itself being a script that runs on top of python3 (you knoow, like pip) - that uses PEP 3107 - Function Annotation as it’s syntax so that a mypy program is fully python 3 compatible. Of course, following PEP 3107 will completely render your code Python 2.x incompatible, but I think the general trend is shifting towards python 3 anyway. Guido recently made some comment on mypy, which for me validated my interest in mypy as something that I should consider with greater attention.

Then today came Armin’s thoughts on optional typing in Python, which I think is NOT an attack on bringing optional typing python but a call to attention on the lack of clear language specification and how this void will brew trouble in implementing an effective optional typing feature. I think Armin’s is more impressed with typed system moving towards a system that is able to infer type - and having a veneer of dynamic types - than on Python which has grown completely unfettered from types chaining itself with a watered down type system.

so..

I do think statically typed languages, combined with type inference, is the sleekest language feature I will be looking out for. But to trying to imagine a complete but optional static type system in python feels un-gestalt - a whole less than the sum of its parts. Mypy does make programming in python a little bit more sloppy and yet to promise any performance gains that is the flagship in statically typed languages (the gap has been narrowed by JIT I belive, but the general consensus still is that dynamically typed languages are slow).

Furthermore, some of the great advantages of dynamically typed languages - best outlined by Laurence Tratt here- will be diluted with an optional type system. One such feature I will miss is the ease of refactoring my python code - a module can easily be refactored with new class names and internal structures given that I don’t remove any functionality the other modules depend on. I will be more than satisfied with python’s code being optionally typed annotated - with no enforcing of the annotation what so ever.
